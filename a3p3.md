# Assignment 3 Part 3: Context Window Extension via Sequence-Length Curriculum Learning

## 1. Introduction

Language models are typically trained at a fixed context length. Nanochat defaults to 2048 tokens, but training at long sequence lengths from the start is expensive — attention cost scales quadratically with sequence length — and can cause instability when the model is poorly initialized (Dataset Decomposition, NeurIPS 2024). An alternative is **sequence-length curriculum learning**: train at a shorter length first to learn local patterns cheaply, then extend to the full length.

We apply this approach to nanochat's picochat model (depth=8, ~42M params) using a two-stage curriculum: Stage 1 trains at sequence length 512 for 3000 steps, producing **Checkpoint 1**. Stage 2 resumes from Checkpoint 1 at sequence length 2048 for 2000 additional steps, producing **Checkpoint 2**. We evaluate both checkpoints at multiple context lengths to measure whether the extension successfully taught the model to leverage longer context.

## 2. Background and Motivation

### 2.1 Sequence-Length Curriculum Learning

Curriculum learning (Bengio et al., 2009) structures training from easy to hard examples. When sequence length serves as the difficulty measure, shorter sequences are "easier" because they require modeling fewer dependencies and are cheaper to compute. Several works validate this for LLM pretraining:

- **GrowLength** (Jin et al., 2023) progressively grows training length from 128 to 4096 tokens, achieving 1.5x faster convergence and 2-3% lower perplexity at the same FLOPs budget. The key insight: "training with shorter sequences in early stages is significantly cheaper per step due to quadratic attention complexity."
- **SkyLadder** (NeurIPS 2025) expands context from 8 to 32K tokens during pretraining, finding that "models pretrained with shorter context windows consistently outperform their long-context counterparts under a fixed token budget." They recommend allocating ~60% of training tokens to the short-context phase — we follow this with our 3000/5000 step split (60%/40%).
- **Dataset Decomposition** (NeurIPS 2024) shows "a correlation between training instability and long sequences, especially at the early stages of training," achieving >2x data efficiency with a short-to-long curriculum.
- **Cerebras VSL** (March 2025) is our most direct precedent: a two-stage approach with Stage 1 at 512 tokens and Stage 2 at 2048 tokens, tested on GPT-3 125M and 1.3B. They find that "models trained with the VSL training recipe achieve similar evaluation scores as models trained entirely with long sequences." We adopt their exact 512→2048 design.

### 2.2 Why 512 for Stage 1

We chose 512 tokens for Stage 1 for four reasons: (1) Cerebras VSL validated exactly 512→2048 on the same architecture class (GPT); (2) 512 provides sufficient context for sentence-level and paragraph-level dependencies; (3) nanochat's picochat ablation baseline uses `MAX_SEQ_LEN=512`, making Stage 1 directly comparable; (4) 512 is a 4x reduction from 2048, giving meaningful compute savings while retaining enough context for learning local patterns.

### 2.3 Why Context Extension Works Without Architecture Changes

Nanochat uses Rotary Position Embeddings (RoPE) (Su et al., 2021), which encode position via rotation matrices applied to query/key vectors with no learned positional parameters. In nanochat, RoPE buffers are non-persistent and recomputed on model initialization (`gpt.py` lines 195-199), and all weight tensors have no sequence-length dimension. Loading a 512-trained checkpoint at `--max-seq-len=2048` simply recomputes RoPE for the new length — no interpolation or architectural changes are needed.

We reset the optimizer state on extension (`--no-load-optimizer`) because the momentum buffers from the short-context loss landscape are stale for the new regime. This follows standard practice in context extension (Cerebras VSL).

### 2.4 Evaluation Design: Multi-Length BPB

Standard perplexity measured at a single context length can mask whether a model truly leverages long context. Fang et al. (2024) show that standard PPL "overlooks key tokens essential for long-context understanding" and has weak correlation with actual long-context performance. Measuring at multiple context lengths reveals the model's effective context window: a model that uses long context should show **decreasing BPB as context length increases**, since more context provides more information for prediction.

We use BPB (bits per byte) rather than perplexity because BPB is tokenizer-agnostic — it normalizes by bytes rather than tokens, keeping comparisons valid across tokenizations (The Gradient). Nanochat already uses BPB as its primary metric. We evaluate at sequence lengths [128, 256, 512, 1024, 2048] for both checkpoints.

## 3. Method

### 3.1 Two-Stage Training

**Stage 1 → Checkpoint 1.** Train picochat (depth=8, ~42M params) at seq_len=512 for 3000 steps with constant learning rate (`warmdown-ratio=0.0`). The learning rate is kept constant because training continues in Stage 2 — decaying it here would force Stage 2 to restart from an artificially low LR. This stage teaches the model local linguistic patterns: sentence structure, token co-occurrences, and short-range dependencies.

**Stage 2 → Checkpoint 2.** Load Checkpoint 1 and continue training at seq_len=2048 for 2000 more steps with a fresh optimizer (`--no-load-optimizer`) and learning rate warmdown (`warmdown-ratio=0.5`). The fresh optimizer avoids stale momentum from the short-context regime. The warmdown stabilizes training as the model fine-tunes on the longer context. RoPE embeddings recompute automatically for the new sequence length.

### 3.2 Code Changes

Two modifications to nanochat:

1. **`scripts/base_train.py`** (+5 lines): Added `--no-load-optimizer` flag that skips loading optimizer state when resuming, enabling context extension with a fresh optimizer.

2. **`scripts/ctx_eval.py`** (~130 lines): New evaluation script that loads a checkpoint, overrides `sequence_len` in the model config, and measures `val/bpb` at each target sequence length using nanochat's existing `evaluate_bpb()` function.

## 4. Results

### 4.1 Training Dynamics

#### Figure 1: Validation BPB (`val/bpb`)

![val/bpb](figures/val_bpb.png)

During Stage 1 (steps 0–3000, seq_len=512), validation BPB decreases from ~1.5 to ~1.22 as the model learns local patterns. At the Stage 2 transition (step 3000, seq_len=2048), BPB initially spikes — the model is now evaluated on 4x longer sequences containing positions it has never seen. This spike is expected: the RoPE attention scores at positions >512 are initially out-of-distribution. The BPB then drops rapidly, falling below Stage 1's minimum and reaching ~1.17 by step 5000. The fact that validation BPB at 2048 tokens surpasses the Stage 1 minimum measured at 512 tokens confirms the model is leveraging the additional context to make better predictions.

#### Figure 2: Training Loss (`train/loss`)

![train/loss](figures/train_loss.png)

Training loss follows a similar trajectory. The jump at step 3000 reflects two concurrent changes: (1) the model encounters 2048-token sequences with long-range dependencies it hasn't learned, and (2) the optimizer resets, losing all accumulated momentum. The loss decreases throughout Stage 2 but does not return to pre-transition levels — this is expected because the task is inherently harder. Predicting tokens conditioned on 2048 positions requires modeling longer-range dependencies than predicting at 512, so the per-token loss is naturally higher even for a model that is using the context well.

#### Figure 3: Learning Rate Schedule (`train/lrm`)

![train/lrm](figures/train_lrm.png)

Stage 1 uses a constant learning rate (lrm=1.0, `warmdown-ratio=0.0`) because training continues into Stage 2 — decaying the LR prematurely would handicap the extension phase. Stage 2 uses `warmdown-ratio=0.5`, which begins the linear LR decay at 50% of total iterations (step 2500). Since Stage 2 resumes at step 3000, the LR is already in the warmdown phase. This is intentional: the model has a strong foundation from Stage 1, and a decaying LR prevents overshooting as it adapts to the new context length.

### 4.2 Multi-Length BPB Evaluation

| seq_len | Checkpoint 1 (step 3000) | Checkpoint 2 (step 5000) | Change |
|---------|--------------------------|--------------------------|--------|
| 128     | 1.2447                   | 1.1964                   | -3.9%  |
| 256     | 1.2253                   | 1.1748                   | -4.1%  |
| 512     | 1.2201                   | 1.1655                   | -4.5%  |
| 1024    | 1.2483                   | 1.1606                   | -7.0%  |
| 2048    | 1.3527                   | 1.1675                   | -13.7% |

#### Checkpoint 1: Context ceiling at 512

Checkpoint 1's BPB decreases from 128→512 (1.245→1.220), confirming the model learned to condition on up to 512 tokens. Beyond that, BPB degrades: +0.028 at 1024 and +0.133 at 2048. This is the expected failure mode for RoPE-based models evaluated beyond their training length. LongRoPE (Ding et al., 2024) describes this precisely: "extrapolating positional encodings beyond the trained range results in unstable and out-of-distribution attention scores." The model attends to positions it never learned to interpret, corrupting rather than improving predictions.

#### Checkpoint 2: Successful context extension

Checkpoint 2's BPB decreases monotonically from 128→1024 (1.196→1.161), a 2.9% improvement from leveraging longer context. This confirms the model learned long-range dependencies during Stage 2 that Checkpoint 1 could not use. The slight uptick at 2048 (1.168 vs 1.161 at 1024) is within normal range — even well-trained models show diminishing returns at the far end of their context window because the marginal information value of very distant tokens is small.

#### Short-context quality preserved

Checkpoint 2 is better than Checkpoint 1 at *every* length, including short contexts: 3.9% better at 128, 4.5% better at 512. This is important because context extension can degrade short-context performance if the model "forgets" local patterns. ProLong (Gao et al., 2024) found that "training only on long data hurts long-context performance" and recommends exposure to both short and long data. Our curriculum provides this naturally: Stage 1 builds the short-context foundation, and Stage 2 trains on 2048-token sequences that still contain short-range patterns within them. The improvement at short contexts is attributable to the additional 2000 training steps.

#### Comparison to Cerebras VSL

Our results align with Cerebras VSL's findings. They observed that Stage-1 models (512-only) "only learn to attend to positions up to the short sequence length" — for the first ~700 tokens both models had similar attention patterns, after which the short model deviated. Our BPB degradation at 1024/2048 for Checkpoint 1 is the loss-based manifestation of this same attention pattern breakdown. The 13.7% BPB improvement at 2048 after Stage 2 confirms that continued training at the target length successfully extends the model's effective context window.

### 4.3 Passkey Retrieval (Needle-in-a-Haystack)

BPB measures compression quality — whether additional context reduces prediction uncertainty on average — but does not test whether the model can retrieve *specific* information from distant positions. Passkey retrieval (needle-in-a-haystack) is a standard task-based evaluation for long-context models: hide a random 5-digit number in filler text, then check if the model can predict it when referenced later. Meta used this methodology to validate Llama 3's context extension (Grattafiori et al., 2024).

Since picochat is a base model (no instruction tuning), we cannot prompt it with questions. Instead, we use a natural repeated-fact pattern compatible with next-token prediction:

```
<BOS>The special number is 47832. [filler sentences...] The special number mentioned above is 47832.
```

The model must predict the passkey digits to complete the restatement. A trial is "correct" only if all passkey tokens are predicted exactly. We run 50 trials per sequence length with deterministic seeding.

| seq_len | Checkpoint 1 (step 3000) | Checkpoint 2 (step 5000) |
|---------|--------------------------|--------------------------|
| 256     | 14%                      | 80%                      |
| 512     | 0%                       | 36%                      |
| 1024    | 0%                       | 0%                       |
| 2048    | 0%                       | 0%                       |

#### Checkpoint 1: No retrieval beyond 256

Checkpoint 1 achieves only 14% accuracy at seq_len=256 and 0% at all longer lengths. Even at 256 tokens — well within its 512-token training length — the model rarely retrieves the passkey. This indicates that 3000 steps of training at seq_len=512 is sufficient for compression (BPB decreases through 512 in Section 4.2) but not for precise information retrieval. The model learns statistical patterns that reduce average prediction uncertainty without learning to attend to and copy specific tokens from earlier in the context.

#### Checkpoint 2: Improved but limited retrieval

Checkpoint 2 shows a substantial improvement: 80% accuracy at 256 and 36% at 512. The additional 2000 training steps at seq_len=2048 not only extended the context window for compression but also improved the model's ability to retrieve specific information at short contexts. However, accuracy drops to 0% at 1024 and 2048, indicating that while compression extends to these lengths (BPB improves through 2048), precise token-level retrieval does not.

#### Compression vs. retrieval

The contrast between BPB and passkey results reveals two distinct capabilities. BPB measures whether additional context reduces average prediction uncertainty — a "soft" use of context where the model extracts statistical signal from distant tokens. Passkey retrieval measures whether the model can locate and reproduce specific information — a "hard" use requiring precise attention to a particular position. Picochat at 42M parameters achieves the former but not the latter at long contexts, which is consistent with findings that needle-in-a-haystack performance scales with model size (Grattafiori et al., 2024). Larger models with more attention heads can maintain sharper attention patterns over longer distances.

## 5. Reproduction

### Run on Modal

```bash
# Full pipeline (data + tokenizer + Stage 1 + Stage 2 + evals)
uv run modal run ctx_modal.py::main

# Or individual stages
uv run modal run ctx_modal.py::run_ctx_stage1
uv run modal run ctx_modal.py::run_ctx_stage2
uv run modal run ctx_modal.py::run_ctx_evals
uv run modal run ctx_modal.py::run_passkey_evals
```

### Run locally

```bash
# Stage 1 → Checkpoint 1
python -m scripts.base_train \
    --depth=8 --head-dim=64 --window-pattern=L \
    --max-seq-len=512 --device-batch-size=16 --total-batch-size=16384 \
    --num-iterations=3000 --warmdown-ratio=0.0 \
    --eval-every=100 --eval-tokens=524288 \
    --core-metric-every=-1 --sample-every=-1 --save-every=500 \
    --model-tag=picochat-ctx-s1 --run=picochat-ctx-stage1

# Stage 2 → Checkpoint 2
python -m scripts.base_train \
    --depth=8 --head-dim=64 --window-pattern=L \
    --max-seq-len=2048 --device-batch-size=4 --total-batch-size=16384 \
    --num-iterations=5000 --warmdown-ratio=0.5 \
    --eval-every=100 --eval-tokens=524288 \
    --core-metric-every=-1 --sample-every=-1 --save-every=500 \
    --resume-from-step=3000 --no-load-optimizer \
    --model-tag=picochat-ctx-s1 --run=picochat-ctx-stage2

# Eval Checkpoint 1
python -m scripts.ctx_eval --model-tag=picochat-ctx-s1 --step=3000 \
    --seq-lens=128,256,512,1024,2048 --device-batch-size=2 --run=ctx-eval-checkpoint1

# Eval Checkpoint 2
python -m scripts.ctx_eval --model-tag=picochat-ctx-s1 --step=5000 \
    --seq-lens=128,256,512,1024,2048 --device-batch-size=2 --run=ctx-eval-checkpoint2

# Passkey eval Checkpoint 1
python -m scripts.passkey_eval --model-tag=picochat-ctx-s1 --step=3000 \
    --seq-lens=256,512,1024,2048 --num-trials=50 --run=passkey-eval-checkpoint1

# Passkey eval Checkpoint 2
python -m scripts.passkey_eval --model-tag=picochat-ctx-s1 --step=5000 \
    --seq-lens=256,512,1024,2048 --num-trials=50 --run=passkey-eval-checkpoint2
```
